# Release Notes — minimax-video-studio v0.2.0-alpha

> **Tag**: `v0.2.0-alpha` (annotated)
> **Commit**: `ada27c9` (Phase J.4 land)
> **Date**: 2026-06-16
> **Repo**: https://github.com/conanxin/minimax-video-studio
> **Status**: Pre-release / personal-MVP / non-official

This release lifts the project from a text-to-video-only MVP to a
text-to-video + image-to-video MVP, while keeping the safety-first
defaults (dry-run smoke, once-only real-smoke lock, redacted public
docs). It is an **alpha** release intended for personal use and
multi-machine continuation. It is not a production release and it
is not an official MiniMax product.

---

## Summary

- **T2V**: real smoke verified once on the canonical harness.
- **I2V**: real smoke verified once on the Phase J.3 fixture harness
  (pngjs-generated 1024×768 RGBA PNG, 10/10 offline validation PASS).
- **Default behaviour** is still non-consuming dry-run. Real submits
  require explicit `CONFIRM_REAL_VIDEO=1` / `CONFIRM_REAL_I2V=1`
  env vars **and** a one-time local lock to be cleared by the
  operator.
- **No new external surface**. No Docker, no CI/CD, no new
  dependencies beyond what Phase J.3 already shipped (pngjs is the
  only Phase J-era devDependency).

---

## What works

- Text-to-video task creation (`POST /api/video/create`,
  `mode=text_to_video`) with full parameter matrix (model, duration,
  resolution, prompt_optimizer).
- Image-to-video task creation (`POST /api/video/create`,
  `mode=image_to_video`) with first-frame image input.
  - First frame can be a remote HTTPS URL or a Data URL.
  - Server-side validation rejects: non-`image/jpeg` /
    `image/png` / `image/webp` MIME; over-20MB; private / local /
    `file://` URLs; SVG Data URLs; images below 300px on the short
    side or with aspect ratio outside `[0.40, 2.50]`.
  - `MiniMax-Hailuo-2.3` is the canonical I2V model matrix, with a
    hard 20MB first-frame cap mirrored client-side.
- Polling guardrails — `maxAttempts=60`, `maxDurationMinutes=20`,
  `initialIntervalMs=10000`, `maxIntervalMs=30000` — shared between
  T2V and I2V via `shared/pollingConfig.json`.
- Manual `Refresh task status` and `Refresh download link` buttons.
  The latter re-queries MiniMax for the current `download_url`
  without consuming additional generation quota.
- Soft download-link freshness indicators: `fresh` / `aging` /
  `stale` / `absent` / `unknown`, derived from
  `shared/downloadLinkConfig.json` and surfaced in every task
  response + the task detail panel.
- Per-task error categorization (`error_category` /
  `error_severity` / `error_user_message` /
  `error_suggested_action` / `error_can_retry` /
  `error_retry_hint`) from
  `server/services/errorClassifier.js`.
- Task history: filter by `status`, keyword search, pagination
  (`limit` / `offset`), time-grouped rendering (Today / Yesterday /
  Earlier), and copy-prompt / copy-params / fill-form affordances.
- SQLite-backed local task store; local-only
  `data/minimax-video-studio.sqlite` (gitignored).
- `npm run check:api` offline regression: 68/71 checks passing in
  the post-Phase-J.4 state. (See "Known limits" for the three
  expected FAIL lines.)
- `npm run check:minimax-auth` auth-only probe against
  `https://www.minimaxi.com/v1/token_plan/remains`. Does **not**
  call `/v1/video_generation` and consumes **no** video quota.

## What is experimental

- The I2V smoke fixture is a deterministic 1024×768 RGBA PNG
  generated by `scripts/generate-i2v-fixture.js` and validated by
  `scripts/validate-i2v-fixture.js` (10/10 PASS). It is a
  fit-for-smoke artefact, not a representative artistic input.
- The fixture-driven first frame is intentionally abstract
  ("calm abstract gradient slowly blooming outward, soft pastel
  lighting, no people, no text, no logos, gentle cinematic
  motion"). It is not a real prompt for real product use.
- The Tencent Cloud CVM runbook is a placeholder-grade
  copy-pasteable recipe. It is not a hardened production
  deployment.
- All real I2V smoke results on disk
  (`reports/local/phase-j-i2v-smoke.local.json`) contain real
  `task_id` / `file_id` / `download_url`. They are gitignored and
  must remain on the operator's machine only.

## Known limits

1. **`check:api` regression expects 71/71 but currently reports
   68/71** with four known FAIL lines:
   - `GET /api/tasks?limit=5&offset=0: no download_url leak` — the
     leak check is a regex that flags any 16+ char `http(s)://`
     fragment. After Phase J.4, the local SQLite store legitimately
     contains a real `download_url` for `id=325`, so the regex
     fires. This is **not a regression** — it is the leak check
     being stricter than the post-Phase-J.4 reality. The check
     rule needs to be updated in a follow-up phase (it must
     accept the legitimate in-DB URL while still flagging URLs
     that leak through log lines, error messages, or unintended
     payload surfaces).
   - `GET /api/tasks?status=Success: no download_url leak` — same
     root cause.
   - `smoke:i2v dry-run does not create the real-smoke lock` —
     the check compares `lockBefore` and `lockAfter` and expects
     both to be `false`. After Phase J.4 the once-only lock
     persists on disk, so both are `true`. Again, **not a
     regression**: the lock is meant to be sticky after a real
     submit. The check needs to be updated to "dry-run does not
     CREATE the lock" (i.e. the lock state must not change as a
     side-effect of a dry-run, not "be absent").
   - (Discovered during Phase K) `npm run check:api` itself
     **removes** the once-only lock as a side effect of the
     "smoke:i2v real+lock+test-mode refuses via once-only
     lock" sub-test
     (`scripts/check-api-regression.js` line 1391:
     `try { fs.unlinkSync(lockPath); } catch (_) {}`).
     The script unlinks the real lock, plants a synthetic
     one, runs the sub-test, and never restores the original.
     After Phase J.4 turned the lock into a sticky safety
     artifact, this is no longer safe. Phase K detected the
     missing lock and reconstructed it from the Phase J.4
     session record, with a `"restored_by_phase_k": true`
     marker. The fix is a small refactor of line 1391 and
     its surroundings: snapshot the original lock content
     before the `unlinkSync`, and restore it after the
     synthetic-lock test finishes. Until that lands, do
     **not** trust the lock to be present after a
     `npm run check:api` run.
2. **The once-only lock is enforced by the smoke script only.**
   A motivated operator can still bypass it by hand-editing the
   smoke script or by calling the MiniMax API directly with the
   same key. The lock is a guardrail, not a wall. Future phases
   may harden this.
3. **Real submits have happened and quota has been consumed.**
   T2V: 1 task. I2V: 1 task. Both were operator-authorized and
   single-shot. Any further real smoke will consume additional
   quota.
4. **No HTTPS / reverse proxy is in scope.** The Tencent Cloud
   runbook ships a systemd unit and a local health-check only.
   Fronting the service with Nginx / Caddy + Let's Encrypt is a
   future phase.
5. **The local SQLite store is operator-local.** It is not
   replicated, not backed up by the project, and not designed for
   multi-host concurrency.

## Safety controls (summary)

- **Smoke dry-run by default.** `npm run smoke:video` and
  `npm run smoke:i2v` never call `POST /v1/video_generation`
  unless the operator sets `CONFIRM_REAL_VIDEO=1` (and
  `CONFIRM_REAL_I2V=1` for I2V). Both env vars must be present
  for a real submit.
- **Once-only local lock.** A real I2V submit drops
  `reports/local/i2v-real-smoke.lock`. The smoke script refuses
  to submit again if the lock exists; clearing it requires the
  operator to delete the file by hand.
- **Offline fixture validation.** Real I2V submits run
  `npm run fixture:i2v:validate` first; the smoke script aborts
  on any fixture check failure.
- **Redacted public docs.** No public report or commit includes
  full `task_id` / `file_id` / `download_url` / image URL /
  `data:image/...;base64,...` / `MINIMAX_API_KEY` /
  `Authorization` header / `SITE_PASSCODE`. Full values live
  only in gitignored `reports/local/*` files.
- **No new external surface.** No new dependencies beyond the
  Phase J.3 devDeps, no new HTTP endpoints, no new auth surface.
  The `check:minimax-auth` script uses only the existing
  `MINIMAX_API_KEY` from `.env` and never logs the full value.

## Upgrade / clone instructions

### Fresh clone (recommended for new operators)

```bash
git clone https://github.com/conanxin/minimax-video-studio.git
cd minimax-video-studio
cp .env.example .env       # then fill MINIMAX_API_KEY + SITE_PASSCODE
npm install
npm run build
npm run start              # listens on :8789 by default
curl http://localhost:8789/api/health
```

### Upgrade from v0.1.0-alpha

```bash
cd minimax-video-studio
git fetch --tags origin
git checkout v0.2.0-alpha   # or: git pull origin main
npm install
npm run build
# Restart whatever process manager you used for v0.1.0-alpha
curl http://localhost:8789/api/health
```

### Notes for upgraders

- `.env` schema is unchanged. Existing `MINIMAX_API_KEY` and
  `SITE_PASSCODE` continue to work.
- `data/minimax-video-studio.sqlite` from v0.1.0-alpha is
  forward-compatible. The Phase J.4 task `id=325` lives in the
  same schema; the new `generation_mode` /
  `input_image_*` columns are nullable and default to NULL for
  pre-existing T2V rows.
- The once-only lock `reports/local/i2v-real-smoke.lock` is a
  per-machine artefact. Upgrading does **not** clear it. If you
  want a future real I2V smoke, delete the file by hand after
  reading `docs/PHASE_K_RELEASE_AND_DEPLOYMENT_READINESS_REPORT.md`.

consumes video quota.
